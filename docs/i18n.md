# Languages

The board speaks English, Spanish and Portuguese. This is how it does that, and
what you need to know before adding a fourth.

## The one rule

**Ids travel. Words stay home.**

A referee taps *chave de braço* in São Paulo and 腕十字固め in Tokyo, and both
publish the same Nostr event: `"submission": "armbar"`. That is what makes
counting armbars across a tournament mean anything.

So:

- `armbar`, `in-progress`, `technical_foul`, `f1` — **protocol**. Never
  translated, in either direction. `rear_naked_choke` printed on a wall is a bug,
  and a relay receiving `chave de braço` is the same bug pointing the other way.
- `Armbar`, `Chave de braço`, `EN VIVO` — **display**. They live in a catalog and
  never leave the browser.

`scoring.ts` may not contain a single word a human reads. It reports *what
happened* — `{ method: 'advantages', detail: { kind: 'tiedAdvantages', top: 4, bottom: 0 } }` —
and `i18n/outcome.ts` says it out loud. If you catch yourself writing a template
literal with an English word in it inside `lib/`, that is the mistake.

## Adding a language

1. Copy `src/lib/i18n/es.ts` to `src/lib/i18n/<code>.ts` and translate the
   values. Keep the `defineCatalog(...)` wrapper: every translation goes through
   it (English is the source and defines itself, so it skips it).
2. Register it in `src/lib/i18n/index.ts`: the code goes in `LOCALES`, the
   catalog in `CATALOGS`.
3. Add `language.<code>` to **every** catalog. A language always names itself
   (`Português`, never `Portuguese`): somebody hunting for their own language
   cannot be asked to read the one the menu happens to be in — that is the very
   problem they are trying to solve.
4. Run `npm run check` and `npx vitest run`.

You cannot forget a key. `Catalog` is a mapped type over the English catalog, so
a missing message is a compile error that names it:

```
Type '{ ... }' is missing the following properties from type 'Catalog':
"app.description", "header.toggleTheme", ... and 45 more.
```

And you cannot forget a *parameter*, which the type alone would have let you:
TypeScript accepts a function taking fewer arguments than the one it stands in
for, so `home.matchCount: () => 'lutas'` would have compiled and put a count with
no number on a wall. `defineCatalog` holds every message to the exact arity of
its English original. That is the entire reason the catalogs are TypeScript and
not JSON.

## Things that are not string replacement

**Plurals.** Never `n !== 1 ? 's' : ''`. A message that carries a count is a
function, and it asks `Intl.PluralRules` for the form:

```ts
'home.matchCount': (count: number) =>
  plural('pl', count, { one: `${count} walka`, few: `${count} walki`, other: `${count} walk` })
```

English and Spanish have two forms, Polish three, Arabic six. `other` is
required; supply whichever others your language actually uses.

**And check what your language does with zero**, because CLDR may not agree with
what people write. `Intl.PluralRules('pt').select(0)` answers `one` — Portuguese
files zero under the singular — so the rules alone print `0 luta`, and no
Brazilian gym writes that. A `zero` form **overrides** the CLDR category when the
count is 0:

```ts
'home.matchCount': (count: number) =>
  plural('pt', count, {
    zero: `${count} lutas`,   // CLDR would have said `one` here
    one: `${count} luta`,
    other: `${count} lutas`
  })
```

English and Spanish declare none, and their zero goes through `other` as usual.
This is the empty screen — an organizer connects and the board counts what it has
before the first match arrives — so it is the first thing anybody sees.

**Values go inside messages, never around them.** `${f1} vs ${f2}` is one message
taking two fighters, not three strings glued together — a language may want them
in the other order. Same for the dash in `four penalties — knee reap`: even the
punctuation is a choice a language gets to make.

**The referee's own words are never translated.** An unknown submission
(`baratoplata`) and a dq detail (`knee reap`) are free text and reach the wall
exactly as they were typed. A board that blanked on a technique nobody has a key
for would be hiding the most interesting thing that happened all day.

## The wall has a width

Some of these messages sit in a fixed pill on a broadcast screen read from across
a gym, or under a number in a column three of them share. English says `LIVE` in
four letters; Portuguese says `AGUARDANDO`. A word that is merely *long* passes
the type check, passes the build, and then quietly runs under the fighters' names
in a room full of people.

So there is a budget, and it is a test — `i18n.test.ts`, *"the words fit on the
wall"*:

| where | keys | budget |
|---|---|---|
| the status pill on the broadcast wall | `status.*` | 12 characters |
| the winner banner | `method.*` | 18 |
| the point breakdown, three across | `score.pt*` | 6 |
| the ADV / PEN pills on the wall | `score.advantages`, `score.penalties` | 5 |
| the chips beside a number on a card | `score.advantageShort`, `score.penaltyShort` | 2 |

It runs for every locale, and it grows on its own as locales are added. If your
language genuinely needs more room, **change the layout** — do not raise the
number and hope. The test failing is how that conversation starts, instead of a
photo of a broken scoreboard arriving after the tournament.

## Two deliberate omissions

**No `Intl.NumberFormat`.** Scores are integers between 0 and about 20, and every
locale shipped renders them identically. A formatter would be ceremony with no
output. When a locale that shapes digits differently shows up, the score messages
(`outcome.score`) are already functions — that is where it goes, and nothing else
has to move.

**No language in the URL.** The language is a setting (`localStorage`, key
`choke:locale`), detected from `navigator.language` on the first visit. This is a
static single-page app read off a wall behind an organizer's pubkey; per-language
URLs would buy indexing nobody is asking for, at the cost of a prerender pass per
locale and a prefix on every internal link.
